Why Z

Wrong changes look correct until something breaks

Agents edit whole repos in one pass. A bad diff often looks identical to a good one until CI or production fails. Z makes the missing evidence visible before you ship — and won't treat a failed suite as “no tests.”

01 — Evidence

Facts before judgment

Exit codes, discovered tests, README sections, symbols — not model vibes or fake confidence %.

02 — Gate

Verify before commit

High blockers hard-stop. Medium needs an explicit ack. Process noise never invents product features.

03 — Memory

Skills you accept

Playbooks grounded in real diffs. Draft until you accept — wrong skills stay quarantined.

How it works

Typed requirements, then proof

Every task becomes a checklist. Each line is classified — product, process, verification, documentation, quality — and scored only against the evidence that kind accepts.

01

Decompose the ask

“Run the tests” is verification. “Don’t commit until verified” is process. “Document in README” is documentation — not a product feature hunt.

02

Bind the right proof

Product needs symbols and behavioral tests. Process needs the session verify log. Docs need the README section. No cross-wiring.

03

Gate, then commit

Suite runs. Detectors fire. At most a few blockers surface. You review with /uncertainties — then the tree can clear.

Skills

Teach the repo once

Coding agents forget your project’s rules when the chat ends. Capture how this repo actually works — grounded in the files and symbols that changed — and Z routes the right playbook on the next matching task.

01 — Capture

From a real turn

Paste with /skills add, generate with z skill create, or save after a solid edit. Captures start as draft.

02 — Accept

You promote what fits

Draft and rejected skills never auto-apply. Review with z skill show, then z skill accept when it’s right.

03 — Route

Matched by repo signals

On a later task, Z scores languages, paths, and intent — not just a title — so an unrelated playbook stays out of the way.

Commit gate

What blocks a commit

Risk and confidence stay separate. High means hard stop. Process and decision gaps stay informational — they never invent product work.

High

Hard block

Failed or zero tests. Unfinished core product requirements. Verification never meaningfully passed. Ignore does not clear these.

Medium

Explicit ack

Partial gaps, API assumptions, migrations, edge cases — you acknowledge before the commit lands.

Low

Visible, not blocking

Process rules already proven by the session, scaffolding noise, and notes you can browse later.

What gets flagged

Untested path Missing co-located tests — or suite failed. Never “no tests” when seven actually ran.
Requirement gap Evidence-bound: code for product, verify log for process, README for docs.
Edge case blind spot Structural branches (else / except / None) undiscussed and untested.
Unverified assumption External API or MCP result without a live check this session.
High-stakes surface Payment, auth, security, data-loss paths — raised even when confidence is high.
Migration risk Schema changes without a clear existing-data impact story.
Integration ripple Widely referenced symbols changed — blast radius above threshold.
Evidence of safety Positive signal when a change matches a tested pattern and the suite passes.
